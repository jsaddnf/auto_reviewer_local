FIELD GUIDANCE:

issues
  Concrete bugs, vulnerabilities, performance concerns, or correctness gaps at specific line numbers.

impact.modified_symbols
  The actual functions/methods/classes/types/constants that the diff touches. Reviewers use this to know "which code paths this commit walks." Mark `signature_changed` whenever the public shape changes (params, return type, throws, protocol additions) — those imply caller updates. `config` is for tuning constants, defaults, ExpManager values, feature flags.

impact.affected_callers
  Real call sites you found by grepping the codebase. For each, name the caller (the function/method making the call) and explain how the change ripples to them (e.g., "must update arg X", "now subject to new throws", "default behavior changed under it"). Empty array if you cannot search.

impact.behavioral_changes
  Higher-level shifts that don't map to a single symbol — e.g. "thread-safety contract weakened on X", "default value shifted from 1.5 to 3 for downstream consumers", "new state Y added to enum Z, breaking exhaustive switches", "retry policy now exponential, slower failure path", "logging silenced on early-return branch".

tests
  Concrete test cases the author should add — specific enough that a developer can implement them from your description alone. Prefer:
  - regression tests for any real bug listed in `issues`
  - edge cases the diff introduces or relaxes (boundary values, error paths, threading)
  - contract tests when `signature_changed` symbols exist
  Skip generic "happy path covered" entries; the `why` field must justify why this test catches something other tests would miss.

SEVERITY GUIDANCE:
- "ok": no issues at all
- "low": style, naming, minor maintenance
- "medium": likely bugs, performance concerns, missing tests on important logic
- "high": security vulnerabilities, data corruption risks, correctness bugs in critical paths, breaking changes
